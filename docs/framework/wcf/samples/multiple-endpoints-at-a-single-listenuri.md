## <a name="service"></a><span data-ttu-id="dcb2b-115">Service</span><span class="sxs-lookup"><span data-stu-id="dcb2b-115">Service</span></span>  
 <span data-ttu-id="dcb2b-116">Il servizio in questo esempio ha due contratti, `ICalculator` e `IEcho`.</span><span class="sxs-lookup"><span data-stu-id="dcb2b-116">The service in this sample has two contracts, `ICalculator` and `IEcho`.</span></span> <span data-ttu-id="dcb2b-117">Oltre all'endpoint `IMetadataExchange` consueto, sono tre gli endpoint dell'applicazione, come mostra il codice seguente.</span><span class="sxs-lookup"><span data-stu-id="dcb2b-117">In addition to the customary `IMetadataExchange` endpoint, there are three application endpoints, as shown in the following code.</span></span>  
  
```xml  
<endpoint address="urn:Stuff"  
        binding="wsHttpBinding"  
        contract="Microsoft.ServiceModel.Samples.ICalculator"
        listenUri="http://localhost/servicemodelsamples/service.svc" />  
<endpoint address="urn:Stuff"  
        binding="wsHttpBinding"  
        contract="Microsoft.ServiceModel.Samples.IEcho"
        listenUri="http://localhost/servicemodelsamples/service.svc" />  
<endpoint address="urn:OtherEcho"  
        binding="wsHttpBinding"  
        contract="Microsoft.ServiceModel.Samples.IEcho"
        listenUri="http://localhost/servicemodelsamples/service.svc" />

 <span data-ttu-id="dcb2b-118">Tutti e tre gli endpoint sono ospitati sullo stesso `ListenUri` e utilizzano la stessa `binding`- endpoint sullo stesso `ListenUri` devono avere la stessa associazione, perché condividono un solo stack di canali che ascolta i messaggi su quell'indirizzo fisico del computer.</span><span class="sxs-lookup"><span data-stu-id="dcb2b-118">All three endpoints are hosted at the same `ListenUri` and use the same `binding` - endpoints at the same `ListenUri` must have the same binding, because they are sharing a single channel stack that listens for messages at that physical address on the machine.</span></span> <span data-ttu-id="dcb2b-119">L' `address` di ogni endpoint è un URN; sebbene in genere gli indirizzi rappresentano posizioni fisiche, infatti l'indirizzo può essere qualsiasi tipo di URI, perché l'indirizzo viene utilizzato per far corrispondere e filtrare scopi come è dimostrato in questo esempio.</span><span class="sxs-lookup"><span data-stu-id="dcb2b-119">The `address` of each endpoint is a URN; though typically addresses represent physical locations, in fact the address can be any kind of URI, because the address is used for matching and filtering purposes as is demonstrated in this sample.</span></span>  
  
 <span data-ttu-id="dcb2b-120">Poiché tutti e tre gli endpoint condividono lo stesso `ListenUri` , quando arriva un messaggio, Windows Communication Foundation (WCF) deve decidere a quale endpoint è destinato il messaggio.</span><span class="sxs-lookup"><span data-stu-id="dcb2b-120">Because all three endpoints share the same `ListenUri`, when a message arrives there, Windows Communication data-stu-id="dcb2b-122">The address filter matches the `To` of the SOAP message to the address of the service endpoint.</span></span> <span data-ttu-id="dcb2b-123">Ad esempio, solo i messaggi con indirizzo `To "Urn:OtherEcho"` sono candidati per il terzo endpoint di questo servizio.</span><span class="sxs-lookup"><span data-stu-id="dcb2b-123">For example, only messages addressed `To "Urn:OtherEcho"` are candidates for the third endpoint of this service.</span></span> <span data-ttu-id="dcb2b-124">Il filtro del contratto associa le azioni associate alle operazioni di un data-ttu-id="dcb2b-126">`Echo` fa corrispondere i filtri del contratto del secondo e del terzo endpoint di questo servizio, perché entrambi quegli endpoint ospitano il contratto `IEcho`.</span><span  <span data-ttu-id="dcb2b-127">Così la combinazione di filtro dell'indirizzo e filtro del contratto rende possibile il routing di ogni messaggio che arriva al `ListenUri` di questo servizio all'endpoint corretto.</span><span class="sxs-lookup"><span data-stu-id="dcb2b-127">Thus the combination of address filter and contract filter makes it possible to route each message that arrives at this service's `ListenUri` to the correct endpoint.</span></span> <span data-ttu-id="dcb2b-128">Il terzo endpoint è differente dagli altri due perché accetta messaggi inviati a un indirizzo diverso dagli altri endpoint.</span><span class="sxs-lookup"><span data-stu-id="dcb2b-128">The third endpoint is differentiated from the other two because it accepts messages sent to a different address from the other endpoints.</span></span> <span data-ttu-id="dcb2b-129">I primo e secondo endpoint sono diversi uno dall'altro sulla base dei contratti (azione del messaggio in arrivo).</span><span class="sxs-lookup"><span data-stu-id="dcb2b-129">The first and second endpoints are differentiated from each other based on their contracts (the Action of the incoming message).</span></span>  
  
## <a name="client"></a><span data-ttu-id="dcb2b-130">Client</span><span class="sxs-lookup"><span data-stu-id="dcb2b-130">Client</span></span>  
 <span data-ttu-id="dcb2b-131">Nel momento in cui gli endpoint del server hanno due indirizzi diversi, anche gli endpoint client hanno due indirizzi.</span><span class="sxs-lookup"><span data-stu-id="dcb2b-131">Just as endpoints on the server have two different addresses, client endpoints also have two addresses.</span></span> <span data-ttu-id="dcb2b-132">Su server e client, viene chiamato l'indirizzo logico `EndpointAddress`.</span><span class="sxs-lookup"><span data-stu-id="dcb2b-132">On both server and client, the logical address is called the `EndpointAddress`.</span></span> <span data-ttu-id="dcb2b-133">Ma mentre l'indirizzo fisico è chiamato `ListenUri` nel server, sul client l'indirizzo fisico è chiamato `Via`.</span><span class="sxs-lookup"><span data-stu-id="dcb2b-133">But whereas the physical address is called the `ListenUri` on the server, on the client, the physical address is called the `Via`.</span></span>  
  
 <span data-ttu-id="dcb2b-134">Come nel server, per impostazione predefinita, questi due indirizzi sono gli stessi.</span><span class="sxs-lookup"><span data-stu-id="dcb2b-134">As on the server, by default, these two addresses are the same.</span></span> <span data-ttu-id="dcb2b-135">Per specificare una `Via` sul client diversa dall'indirizzo dell'endpoint viene utilizzato `ClientViaBehavior`:</span><span class="sxs-lookup"><span data-stu-id="dcb2b-135">To specify a `Via` on the client that is different from the endpoint's address, `ClientViaBehavior` is used:</span></span>

        new ClientViaBehavior(via));  
```  
  
 <span data-ttu-id="dcb2b-136">Come al solito, l'indirizzo proviene dal file di configurazione client, generato da Svcutil.exe.</span><span class="sxs-lookup"><span data-stu-id="dcb2b-136">As usual, the address comes from the client configuration file, which was generated by Svcutil.exe.</span></span> <span data-ttu-id="dcb2b-137">`Via` (che corrisponde a `ListenUri` del servizio) non viene visualizzato nei metadati del servizio e così queste informazioni devono essere comunicate al client fuori banda (proprio come l'indirizzo dei metadati del servizio).</span><span class="sxs-lookup"><span data-stu-id="dcb2b-137">The `Via` (which corresponds to the `ListenUri` of the service) does not appear in the service's metadata and so this information must be communicated to the client out-of-band (just like the service's metadata address).</span></span>  
  
 <span data-ttu-id="dcb2b-138">Il client in questo esempio invia messaggi a ognuno dei tre endpoint dell'applicazione del server, per dimostrare che può comunicare con (e distinguere tra) i tre endpoint, anche se tutti hanno la stessa `Via`.</span><span class="sxs-lookup"><span data-stu-id="dcb2b-138">The client in this sample sends messages to each of the server's three application endpoints, to demonstrate that it can communicate with (and differentiate) all three endpoints, even though they all have the same `Via`.</span></span>
